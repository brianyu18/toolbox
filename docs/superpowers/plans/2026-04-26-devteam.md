# devteam 1.0.0 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Rename `toolbox` plugin (0.1.0) to `devteam` (1.0.0) and rebuild as a multi-agent dev team — LEAD orchestrator skill + 4 dialogue-phase skills (THINKER, PLANNER, SHIPPER, REFLECTOR) + worker agents (BUILDER, review-specialist, TESTER) + 4 utility agents (EXPLORER, CRITIC, SYNTHESIZER, INVESTIGATOR), with a chronological team-slack audit log, two invocation modes (LEAD vs direct), and two autonomy modes (Work-Together vs Autonomous).

**Architecture (final, post all reviews):** LEAD is a skill in the main thread — the only persona the user talks to in LEAD mode. User-dialogue phases (THINK, PLAN, SHIP, REFLECT) are skills loaded into LEAD's turn. Heads-down/parallel phases (BUILD, REVIEW, TEST) are dispatched directly by LEAD via the Task tool — there are NO lead-agent middleware files. LEAD reads PLANNER's wave-grouped partitions (for BUILD), `bin/devteam-pick-lenses.sh` output (for REVIEW), and `bin/devteam-detect-stack.sh --tests` output (for TEST), then fires workers in parallel from one Task message per wave. Workers (BUILDER ×N, review-specialist ×N with lens files, TESTER ×N) run in fresh subagent contexts and return result packets. LEAD merges results, applies diffs serially, writes artifacts. (Why no nested dispatch: Claude Code does not support subagents dispatching subagents; verified via doc + GitHub issue research, plan-eng-review finding A1; outside-voice review found that planning-only lead agents were vestigial transforms — the work folds cleanly into PLANNER + bash detectors + LEAD's own body.) Utility agents (EXPLORER, CRITIC, SYNTHESIZER, INVESTIGATOR) are dispatched from any skill turn. Persistence uses artifact-backed handover: every phase writes to `.devteam/state/` and a per-project chronological `slack.md`. Workers never call AskUserQuestion — they return structured question packets to LEAD, which funnels to the user with both specialist + LEAD recommendations. Backward compat: old `/toolbox-*` commands route to LEAD with `--tier <name>` flags.

**Tech Stack:** Markdown skill/agent/command files + YAML frontmatter (Claude Code plugin spec). POSIX shell (sh) for `bin/slack-append.sh` (mkdir-mutex for race-safe append; no `flock` dependency). JSON for plugin manifest + machine-readable state files. Per-stack convention library at `~/.claude/devteam/conventions/` seeded from bundled `conventions-seed/`.

**Spec:** `docs/superpowers/specs/2026-04-26-devteam-design.md` — 38 locked-in commitments. This plan implements that spec, with adjustments below from plan-eng-review (2026-04-26).

## Resolved decisions from plan-eng-review + plan-ceo-review

These are amendments applied across the plan. When implementing, treat these as overriding any earlier text that contradicts them.

- **A1 (P0) — FINAL via Plan B + outside-voice revision:** Subagents cannot dispatch other subagents in Claude Code (verified). After outside-voice review, the lead-agent layer was identified as vestigial — once leads are planning-only, their work is small deterministic transforms that don't earn agent status. **Final architecture: NO lead agents.** Lead responsibilities split as follows:

  - **REVIEW lens selection** → new bash script `bin/devteam-pick-lenses.sh` (deterministic regex over `git diff --name-only` to return lens list).

  - **Dispatch + merge** for BUILD/REVIEW/TEST → moves into LEAD skill body directly.

  LEAD becomes the hands-on orchestrator: reads PLANNER's wave-grouped partitions / pick-lenses output / detect-stack output, composes worker briefs, dispatches workers in parallel via Task tool, collects results, applies/merges. **Workers are still subagents** (BUILDER, review-specialist, TESTER + 4 utility agents).
```sh
#!/bin/sh
# bin/slack-append.sh "<entry-line>"
# Race-safe append using mkdir-mutex (POSIX-portable; no flock dependency).
set -eu

if [ "$#" -ne 1 ]; then
  echo "usage: $0 \"<entry-line>\"" >&2
  exit 2
fi

STATE_DIR="${DEVTEAM_STATE_DIR:-${PWD}/.devteam/state}"
LOCKDIR="${PWD}/.devteam/slack.lock.d"
SLACK="$STATE_DIR/slack.md"

mkdir -p "$STATE_DIR" "$(dirname "$LOCKDIR")"
touch "$SLACK"

TRIES=0
while ! mkdir "$LOCKDIR" 2>/dev/null; do
  # OV-3: stale-lock detection. If lockdir is older than 30s, presume the previous
  # holder crashed without releasing. Force-remove and retake.
  if [ -d "$LOCKDIR" ]; then
    LOCK_AGE=$(( $(date +%s) - $(stat -f %m "$LOCKDIR" 2>/dev/null || stat -c %Y "$LOCKDIR" 2>/dev/null || echo 0) ))
    if [ "$LOCK_AGE" -gt 30 ]; then
      echo "slack-append: stale lockdir (age ${LOCK_AGE}s) — force-removing" >&2
      rmdir "$LOCKDIR" 2>/dev/null || rm -rf "$LOCKDIR"
      continue
    fi
  fi
  sleep 0.05
  TRIES=$((TRIES+1))
  if [ "$TRIES" -gt 200 ]; then
    echo "slack-append: lock timeout after 10s, forcing append" >&2
    break
  fi
done
trap 'rmdir "$LOCKDIR" 2>/dev/null || true' EXIT INT TERM

# C4: per-actor counter management. Caller's entry-line is expected to contain a
# `[ACTOR#]` placeholder that this script substitutes with the next counter value
# for that actor. Counter files live at $STATE_DIR/.counters/<actor>.
ENTRY_LINE="$1"
if echo "$ENTRY_LINE" | grep -q '\[[^]]*#\]'; then
  ACTOR=$(echo "$ENTRY_LINE" | sed -E 's/.*\[([^]]+)#\].*/\1/')
  CTR_DIR="$STATE_DIR/.counters"
  CTR_FILE="$CTR_DIR/$ACTOR"
  mkdir -p "$CTR_DIR"
  CUR=$(cat "$CTR_FILE" 2>/dev/null || echo 0)
  NXT=$((CUR + 1))
  printf "%d" "$NXT" > "$CTR_FILE"
  ENTRY_LINE=$(echo "$ENTRY_LINE" | sed "s/\[$ACTOR#\]/[$ACTOR#$NXT]/")
fi

TS=$(date -u +"%Y-%m-%d %H:%M:%S")
ID=$(date -u +"%y-%m-%d-%H-%M-%S")
printf "#%s  %s  %s\n" "$ID" "$TS" "$ENTRY_LINE" >> "$SLACK"
```

**Caller convention (per C4):** Pass the entry line with `[ACTOR#]` (empty counter); the script substitutes `[ACTOR#NN]` with the next per-actor counter value. Example:
- Caller passes: `"[BUILDER:fe#] DECISION  Picked react-hook-form"`
- Script writes: `"[BUILDER:fe#7] DECISION  Picked react-hook-form"` (assuming 6 prior entries from BUILDER:fe)

```markdown
# Pine Script Conventions

## Stack overview

Pine Script is TradingView's domain-specific language for indicators and strategies. Compiled and run on TradingView's servers; output is plotted on charts. Versions: v5 and v6 (current). v6 has stricter type checking.

- **Version pragma**: every script starts with `//@version=5` or `//@version=6`.

- Visual chart inspection at known historical events.
- Compare alerts/signals against ground truth from Excel/sheet calculations.

- Repainting (using future-bar data unintentionally) — always validate signals don't change after the bar closes.
- `na` propagation in math — wrap with `nz()` to default to 0.
- Confusing `iff()` (deprecated) with ternary `?:`.
```

```markdown
# React Conventions

## Stack overview

Functional React with hooks, TypeScript-first. SSR via Next.js or framework of choice; client-side state via hooks + context.

## Conventions

- **Functional components only.** No class components in new code.
- **One component per file.** File name matches default export (PascalCase).

- **Don't `useState` with the same shape repeatedly** — define a typed reducer.
- **Don't use raw-HTML render APIs** (the React ones whose name suggests danger, or Vue's `v-html`) — they enable XSS if content isn't sanitized; prefer escaping or a sanitizer (DOMPurify) at the trust boundary.
- **Don't export anonymous default** — name the function for stack traces.

## Test patterns

- React Testing Library, not Enzyme.
- Test behavior (what user sees / does), not implementation.
- Use `screen.getByRole(...)` over `getByTestId` when possible (accessibility-aligned).
- Mock at the boundary (network, not internals).

## Common pitfalls

- Stale closures in `useEffect` / `useCallback` deps.
- Re-renders triggered by inline object/array literals as props (memoize or hoist).
- Missing `key` on list items.

## References
```

```markdown
# Tailwind CSS Conventions

## Stack overview

Utility-first CSS framework. Compose styles inline via class names. v3+ uses JIT for fast, dead-code-free builds.

## Conventions

- **Group classes by category in this order**: layout (flex/grid), positioning (absolute, top), spacing (p, m), sizing (w, h), typography (text, font), color (bg, text), interaction (hover, focus), other.

- **Responsive prefixes mobile-first**: write base styles, then `sm:`, `md:`, `lg:` overrides.
- **Use `@apply` sparingly** — only for true CSS primitives where utilities would be too verbose.

- **Don't fight the design system** with arbitrary values everywhere — extend the config instead.
- **Don't use `important!`** — the cascade is fine.
- **Don't re-implement utilities** in custom CSS — Tailwind's are well-tested.

## Test patterns

Visual regression tests (Percy, Chromatic) catch class-related regressions. Static class strings are robust to typos via type-checking with `tailwindcss-classnames` or `cva`.

## Common pitfalls

- Purge config too aggressive → classes not found in production. Configure `content:` paths to include all template files.
- Dynamic class names get purged. Use full class strings or `cva`.

## References
```

```markdown
# Node.js Backend Conventions

## Stack overview

Node 20+, TypeScript, ESM modules. Web frameworks: Express (legacy), Fastify (preferred for new), Hono (edge/lightweight).

- **TypeScript strict mode on** (`"strict": true` in tsconfig).

- **Don't put business logic in route handlers** — routes wire HTTP to services.
- **Don't catch-and-ignore** errors — log + rethrow or handle explicitly.
- **Don't `process.env.X` deep in code** — read once at startup, pass typed config.
- **Don't use synchronous fs/crypto in request paths** — blocks the event loop.
- **Don't mix promise chains and async/await** in the same function.

## Test patterns

- Vitest or Jest. Vitest preferred for ESM-native projects.
- Test public API of services, not private methods.
- Use real DB (test container or in-memory equivalent) over heavy mocking when possible.
- Integration tests for routes with supertest / fastify-inject.

## Common pitfalls

- Unhandled promise rejections crash modern Node — always `await` or `.catch`.
- `JSON.parse` on user input throws synchronously — wrap or validate.
- `process.exit()` in libraries — leave it to the entry point.

- [Fastify docs](https://fastify.dev/docs/latest/)
- [Pino](https://getpino.io/)
```

```markdown
# Postgres Conventions

## Stack overview

PostgreSQL 14+. Migrations in plain SQL files (versioned), or via tools like Knex/Prisma/Drizzle/Atlas. Connection pooling via PgBouncer or framework-native pool.

## Conventions

- **Snake_case** for table and column names.
- **Plural table names** (`users`, not `user`).
- **`id` BIGSERIAL or UUID primary keys** — pick one per project, stick with it.
- **`created_at`, `updated_at` timestamps** with `DEFAULT now()` and trigger to update.
- **Foreign keys explicit** with `ON DELETE` behavior chosen per relationship.
- **Indexes named `<table>_<columns>_idx`** — predictable and greppable.
- **Migrations are forward-only**; rollbacks are new migrations.

## Anti-patterns

- **Don't use `SELECT *` in application code** — list columns explicitly.
- **Don't index everything** — indexes cost on writes; only add when query plans show seq scans on real data.
- **Don't use `text` everywhere** when constrained types fit (`varchar(N)`, `int`, `boolean`, `timestamp`).
- **Don't put business logic in triggers** unless strong reason — debugging gets hard.
- **Don't run migrations without a backup** — even forward-only.

## Test patterns

- Use a separate test database; reset between tests via transaction rollback or schema reset.
- Test migrations: apply, verify schema, apply next, verify, repeat.
- For complex queries: snapshot the query plan and check for regressions.

## Common pitfalls

- N+1 queries — load related rows eagerly with JOIN or use a dataloader.
- Implicit casts in WHERE clauses prevent index use.
- Long-running transactions block VACUUM and bloat the DB.
- Migrations with `ALTER TABLE` on huge tables can lock for long periods — use zero-downtime patterns.

## References

- [Postgres docs](https://www.postgresql.org/docs/current/)
- [Use The Index, Luke!](https://use-the-index-luke.com/)
```
